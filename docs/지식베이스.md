### 1.3 풀어야 할 문제

이 대시보드는 그로잉업팀이 매일 마주하는 **3가지 운영 통증**을 해결합니다:

1. **다중 채널 매출이 흩어져 있음** — 8개+ 판매 채널 (자사몰/네이버/쿠팡 로켓그로스/쿠팡 로켓배송 벤더/무신사/오프라인/이지웰/오늘의집) 매출을 매번 각 플랫폼 어드민에서 따로 확인해야 했음 → **하나의 시트(공식) + API 자동 수집으로 통합**
2. **광고 의사결정이 사람 감각에 의존** — 네이버/쿠팡/Meta 광고를 각각 보고 ROAS·CTR 추세 추정 → **룰 엔진이 매일 자동으로 증액/축소/소재 교체 신호 생성**
3. **재구매 타이밍 놓침** — 30~60일 경과 고객을 수동으로 추리지 못함 → **CRM 페이지가 매일 갱신, CSV 다운로드 즉시 가능**

| **매출 달성률** | 시트 actual + API 보완 / 월 목표 | 월말 페이스 예측, 채널 재배분 |
| **ROAS** | 매출 / 광고비 (채널별) | 자사몰 300% / 네이버 400% / 쿠팡 500% 목표 대비 증액·축소 |

| **광고 효율 추세** | 7일 vs 14일 CTR/ROAS 변화율 | 소재 피로도 감지 |

### 2.1 핵심 설계 원칙

1. **단일 진실 소스(Single Source of Truth) — 시트 우선**
   - 운영팀이 매일 입력하는 시트가 **공식 매출**
   - API가 수집한 값은 시트가 0인 날짜에만 자동 보완 (`utils/data.py::compute_official_actual()`)
   - 사람 입력 + 자동 수집 = 항상 최신 + 정확

2. **Precompute 우선 로딩 — 빠른 응답**
   - 매일 10시 무거운 계산은 1회만 실행 → Parquet/JSON으로 캐시
   - 대시보드는 캐시만 읽음 (네트워크 의존 X)
   - 페이지 진입 < 1초

3. **AI 환각 없는 룰 엔진**
   - LLM 호출 0건 — 모든 인사이트는 임계값/수식 기반
   - 매번 같은 입력 → 같은 출력 (재현성)

### 3.3 Precompute 파이프라인

`scripts/precompute.py` — 4개 함수가 순차 실행:

| 함수 | 출력 | 용도 |
|------|------|------|
| `precompute_sheet()` | `sheet_daily_sales.parquet` | Google Sheets live CSV → 파싱 → 저장 (3,449행, 12개월) |
| `precompute_home_snapshot()` | `home_snapshot.json` | 4개 기간 × 3브랜드 KPI 사전 계산 |
| `precompute_insights()` | `insights_snapshot.json` | 6개 룰 결과 사전 계산 |
| `precompute_campaigns()` | `meta_campaigns_*.parquet`, `naver_campaigns_daily.parquet` | 캠페인 × 일자 단위 광고 성과 |

`mark_last_updated()` — `last_updated.json`에 타임스탬프 저장. 이 값이 Streamlit `@st.cache_data` 키로 들어가서 **precompute 실행 즉시 캐시 무효화**.

### 3.4 시트 + API 통합 로직 — 공식 매출 계산

`utils/data.py::compute_official_actual()` — 모든 페이지가 같은 공식을 사용:

```
공식 매출 (브랜드, 기간) =
    시트 actual 합산 (모든 채널 — 자사/네이버/쿠팡/무신사/오프라인/이지웰/오늘의집 포함)
  + 시트 actual=0 인 날짜의 API orders + 쿠팡 inbound 매출 (자동 보완)
```

**4곳에서 호출:**
1. `pages/2_💰_매출_분석.py` — 총 매출 KPI
2. `app.py` — 홈 브랜드 카드 + CRITICAL 알림
3. `scripts/precompute.py::precompute_home_snapshot()` — 캐시 빌드
4. `utils/insights.py::_pace_forecast()` — 월말 예측

> 이전에는 페이지마다 다른 공식을 써서 홈 16% vs 매출 분석 18% 같은 불일치가 있었음. `compute_official_actual()` 도입 후 모든 페이지 100% 일치.

### 3.5 시트 누락 의심 셀 자동 검출

매출 분석 일별 표(`pages/2`)에 추가된 검출:
- 시트 actual > 0
- AND API ≥ 시트 × 1.2배
- AND 절대 차이 ≥ 10,000원

→ 📌 마커 표시 + 노란 경고 박스 + 상세 expander
→ 표시값은 시트 그대로 유지 (공식 원칙 보존)
→ 팀이 시트 채워주면 다음 새로고침에 자동 사라짐

## 4. AI / 자동 분석 — 현재 탑재된 기능

### 4.1 LLM 호출 — **없음**

현재 대시보드는 **Claude/GPT 등 LLM API 호출이 0건**입니다. 모든 분석은:
- **수식 기반** (가중 통계, EWMA, Pearson 상관)
- **임계값 기반** (ROAS 300%, 광고비 50만원 등)
- **룰 기반** (경과일 + 평균 페이스 예측)

**철학** (`utils/insights.py:5-8`):
> 추정·의견 X → 측정 가능한 수치만
> 방향성 명확할 때만 (임계값 넘은 경우만 노출)
> 의사결정에 직접 도움되는 액션 포함

→ AI 환각 위험 0%, 매번 같은 입력엔 같은 출력 (재현성).

### 4.2 인사이트 엔진 — 6개 룰 (`utils/insights.py`)

| 함수 | 트리거 조건 | 심각도 | 출력 예시 |
|------|----------|------|---------|
| `_pace_forecast` | 월말 예상 달성률 분류 | good 105%↑ / info 90~105% / warning 60~90% / urgent 60%↓ | "루티니스트 — 이번 달 목표 큰 폭 미달 예상 (16%)" |
| `_top_contributor` | 단일 브랜드 매출 ≥50% | info | "롤라루가 전체 매출의 75% 견인 중 — 의존도 높음" |
| `_worst_channel` | 목표 ≥100만 AND 달성률 <15% | urgent | "롤라루 무신사 달성률 8% — 긴급 점검" |
| `_ad_efficiency` | ROAS ≥300% (증액) / <50% AND 광고비 ≥50만 (손실) | good / urgent | "쿠팡 광고 ROAS 461% — 예산 증액 검토" |
| `_crm_opportunity` | 30~60일 경과 고객 존재 | good ≥100명 / info | "재구매 타이밍 고객 N명 — 예상 매출 X원" |
| `_uncollected_target` | 목표 있음 + 실적 0 | warning | "롤라루 무신사·오프라인 운영 공백 의심" |

`generate_insights()` — 우선순위 정렬 후 상위 6개만 노출.

### 4.3 알림 룰 엔진 — 5개 (`utils/alerts.py`)

| 알림 | 트리거 | 액션 |
|------|------|------|
| **ROAS 3일 추세** | 3일 연속 목표 1.3배 초과 (증액) / 3일 연속 미달 (축소) | 광고비 ±20% |
| **CTR 피로도** | 최근 7일 CTR이 직전 7일 대비 30% 이상 하락 | 소재 교체 |
| **채널 재배분** | 최고 채널 +50%p · 최악 채널 -50%p 격차 | 예산 이동 |
| **재구매 타이밍** | 마지막 주문 30~60일 경과 고객 존재 | CRM 발송 |
| **주문 급감** | 최근 7일 주문수 직전 7일 대비 -20% | 원인 점검 |

### 4.4 캠페인 액션 엔진 (`utils/actions.py`)

THRESHOLDS = {

    "repurchase_cycle_days": 30,

    "ctr_fatigue": -0.30,          # CTR -30% 경고
    "roas_up_ratio": 1.30,         # ROAS 목표×1.3 → 증액
    "roas_down_ratio": 1.0,        # 목표 미달 → 축소

**캠페인 분석 페이지** (`pages/1_📣_광고_분석.py`)에서 자동 분류:
- 🚨 낭비 의심: 광고비 ≥50만 AND ROAS < 목표 × 0.5
- ⚠️ 미달: 광고비 ≥10만 AND ROAS < 목표 × 0.8
- 💰 증액 추천: ROAS ≥ 목표 × 1.3 AND 광고비 ≥10만

### 4.5 가중 통계 예측 모델 (`utils/forecasting.py`)

**`weighted_month_end_forecast()`** — 월말 매출 예측

수학 모델:
```
w_i = exp(-Δt_i / τ)              # 지수 감쇄, τ = 14 / ln(2)
weekday_avg[wd] = Σ(actual_i × w_i) / Σ(w_i)   # 요일별 가중평균
trend = recent_7d / baseline_28d           # 최근 추세 [0.5, 2.0]
forecast(d) = weekday_avg[d.weekday()] × trend
projected_total = actual_so_far + Σ forecast(d for d in remaining)
confidence_band = ±1σ (가중 분산)
```

→ 산술평균보다 정확 (요일/추세/최근성 모두 반영)
→ 매출 분석 페이지의 "🔮 월말 매출 예측" 차트에 표시

### 4.6 캠페인-제품 상관 분석 (`utils/forecasting.py`)

**`campaign_product_correlation()`** — 광고와 제품 매출의 가중 Pearson 상관계수
- 최근 14일 데이터 + 공통 관측일 ≥14일 필요
- 상관 0.5 이상 → 캠페인이 해당 제품 매출 견인 가능성 높음
- → 광고 분석 → 캠페인 → 제품 attribution 매핑 차트

### 4.7 제품명 정규화 (`utils/products.py`)

`PRODUCT_NAME_RULES` — 채널별로 흩어진 SKU 이름을 통일:
- "초경량 러닝조끼" → "러닝조끼"
- "스마트폰 거치 만년다이어리" → "만년다이어리"
- "김똑똑 어린이김 (60봉)" → "김똑똑 어린이김 60봉"

→ 제품 분석에서 같은 제품을 채널 무관하게 합산 가능
→ 이미지 캐시 매칭도 정규화된 이름 사용

`BRAND_RULES` + `classify_product()` — 제품명 → 브랜드 자동 분류 (똑똑/롤라/루티/공통)

## 7. 개발 / 운영 FAQ

### Q1. 새 데이터 소스 추가하려면?
1. `api/{새_소스}.py` 클라이언트 생성
2. `scripts/sync_{새_소스}.py` 동기화 CLI 작성
3. `sync_all.bat` 라인 추가
4. `utils/data.py` merge 함수 추가 (orders/ads 스키마 매핑)

### Q2. 새 페이지 추가하려면?
1. `pages/{N}_{이모지}_{이름}.py` 작성
2. `utils/ui.py::_NAV_PAGES` 리스트에 등록
3. `setup_page()` 호출로 사이드바 자동 노출

### Q3. 캐시가 갱신 안 될 때?
- 5분 TTL 자동 만료 대기, OR
- 사이드바 "🔄 캐시 새로고침" (Cloud), OR
- 로컬: 사이드바 "🔄 전체 API + 시트 갱신" → sync_all.bat 백그라운드

### Q4. Cafe24 토큰 만료되면?
- 자동 갱신: `scripts/sync_cafe24.py`가 매번 만료 체크 후 refresh_token으로 갱신
- 수동: `pages/7_🔌_API_연결.py` Cafe24 탭에서 OAuth 재진행

### Q5. 시트와 API 값이 안 맞으면?
- **매출 분석 일별 표 → 📌 마커 + 노란 경고**: 시트 누락 의심 셀 자동 검출
- **컨텍스트**: 시트는 운영팀 수동 입력 → 정산 타이밍/사람 누락 가능
- **API**: 자동 수집 → 정확하지만 Wing API 등의 createdAt 지연 케이스 있음
- **결론**: 시트가 공식 (표시값 유지), API는 보조 신호

### Q6. 새 임계값 조정하려면?
- `utils/actions.py::THRESHOLDS` 수정
- `utils/metrics.py::TARGET_ROAS` (백분율 매핑)
- 변경 후 git push → Cloud 자동 재배포

### Q7. AI(LLM) 분석을 추가하려면?
- 현재 LLM 호출 0건. 도입 시:
  1. `requirements.txt`에 `anthropic` 또는 `openai` 추가
  2. API 키를 `.streamlit/secrets.toml`에 등록
  3. `utils/insights.py`에 `_llm_summary()` 함수 추가 — Claude/GPT가 KPI를 한국어 요약
  4. 페이지에서 캐싱 (월별 1회 호출 권장 — 비용 관리)

### Q8. 데이터 백업?
- `data/orders.csv`, `data/ads.csv`, `data/precomputed/*` 모두 git에 커밋됨
- GitHub에 매일 commit → 시점별 복원 가능
- Cafe24 토큰만 별도 백업 필요 (`data/cafe24_tokens.json`)
